The client alignment deck (`client_alignment_deck.pptx`, generated by `build_client_deck.py`) is the team-shareable artifact for Labcorp client meetings. The user iterates on it heavily — corrections on visual fidelity (overlapping text, undersized boxes, broken arrows) are the most common feedback. Build defensively.

**Why:** the user explicitly rejected the pandoc-rendered markdown approach with "dont you have any skills that you can find and load to create cleaner boxes etc?" — they want editable PowerPoint shapes the client can edit/move/recolor after receipt, not flat images and not monospace ASCII art.

**How to apply:**

**Construction primitives:**
- Use **python-pptx with native PowerPoint shapes** — `MSO_SHAPE.ROUNDED_RECTANGLE`, `MSO_SHAPE.DIAMOND`, `MSO_CONNECTOR.STRAIGHT`/`ELBOW`, `add_table`. Never ASCII art. Never pandoc → pptx for the deck (pandoc loses shape editability).
- **Calibri font everywhere** (matches the user's pandoc-default-style preference for client deliverables — looks human-authored, not AI-generated).
- **16:9 widescreen** (`Inches(13.333) × Inches(7.5)`).
- **Footer convention** at `y=7.18`: thin grey line + "LIMS Story Generation — Client Alignment" label (left, 8pt italic) + "N / total" slide number (right). **All slide content must fit ABOVE `y=7.18`.** Verify with: `(shp.top + shp.height) / 914400 ≤ 7.18` for every shape.
- **Color semantics, consistent across slides:**
  - Blue (`BOX_FILL` / `BOX_LINE`) = process step / input / inherited
  - Yellow (`SYS_FILL` / `SYS_LINE`) = agent action / decision / highlight
  - Yellow diamond (`DECISION_FILL`) = decision diamond
  - Green (`END_FILL` / `OUT_FILL`) = output / kept / admitted / start
  - Pink (`PARK_FILL` / `PARK_LINE`) = park / fail / reject
  - Light grey (`CODE_FILL`, Consolas font) = code / schema box
  - Light orange (`HUB_FILL`) = central hub (slide 2)
  - Purple (`PHASE_FILL`) = phase markers (slide 4)
- **Numbered step badges** (white digit on filled circle, drawn second so it's on top of the box) for sequential steps. See `add_step_box` helper.

**Content principles:**
- **Show, don't just tell.** Diagrams over prose where possible. Each phase gets a flowchart with decision diamonds; each schema gets a YAML example with field table; each abstract concept (e.g., "≥ 2 SOPs lift") gets a worked-example visual.
- **One example per concept that can be exemplified.** 4 story shapes → 4 example cards. Validator → walkthrough of an actual story going through the checks. Dependency Resolver → DAG with 5 stories.
- **Show parallelism visually:** quorum panel rendered as 5 parallel small boxes (one per prompt frame), not a single "5 reviewers" box. Per-SOP pipeline wrapped in a dashed `ForEach SOP` container with a label hugging the top border.
- **YAML schemas at 8.5pt** to fit standard code-box bounds. 9.5pt overflows for typical 28-line YAML in a 4.0" tall box; the overflow visually obscures any bottom diagrams.
- **Specific terminology:** "SOPs" not "documents"; "discipline" not "lab" (see `feedback_terminology.md`).

**Workflow:**
- The deck is regenerated by `python3 build_client_deck.py`. The Python script is the source of truth; the `.pptx` is a build artifact, but commit both (git history then shows iteration cleanly).
- Verify slide bounds after every regeneration before committing.
- The deck has grown over iterations (started 3 slides, now 12). Slide order matters narratively; don't renumber casually.
- Common deck audit: search for "lab" (should only be in "lab_stage_v1" / "lab procedure" / similar; never "new lab" / "similar lab"); search for "document(s)" (should be SOPs); search for unexplained acronyms (the user pushed back on "AC" — spell out at first use).

**Bug pattern to watch for:** elbow connectors with same start.x and end.x degenerate to straight vertical lines pointing to empty space (slides 5 and 7 had this — inherit-branch and keep-concrete-branch arrows ended below the side-branch box rather than reaching the final node). Always make sure connector endpoints land **on a real shape's edge**, not in empty space alongside it.
